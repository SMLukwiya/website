Vaulten-Aura Runtime alpha1

A self-hosted, high performance, edge-native runtime and orchestrator.

Built from scratch with love in my heart and a song on my libs! This platform targets immutability, security, and operator control — without a managed control plane.

Released one year after the first successful segfault-free hello world in C. See inspiration

Why this exists

Edge computing promises low latency, locality and a bright future, but most platforms hide critical decisions behind opaque control planes and generic servers.

Vaulten aura exists to explore a different approach: an edge system where performance, scheduling, and lifecycle decisions are explicit, observable, and under operator control.

This project is the result of the excitement from printing "Hello World" like a code master!, several months spent learning how servers, kernels, I/O, and runtimes actually work.

I could have been a AI developer, but my PC starts coughing blood when I open google chrome and two vscode instances, so that was a sign from above!

What works today (alpha1)

Runtime, orchestration, and control plane

  • Custom edge-native server written in C
  • HTTP/2 only, running over TLS 1.3
  • TLS implemented using picotls and minSSL
  • Pure epoll-based event loop with explicit load control
  • Built-in orchestration of function instances
  • Local, load-aware scaling without a central scheduler
  • CLI to start/stop the daemon and server
  • YAML-based server and function configuration
  • Streaming, event-driven YAML parser with early termination
  • Deep config validation with precise line/column errors
  • Function deployment and invocation (verified via nghttp2)
  • Radix tree–based routing for deployed and active functions
  • Custom binary blob format combining function config and code
  • O(1) access to blob sections via precomputed tables
  • And bugs so deadly that of they decide to arrest me, I won't even complain.

What “edge-native” means in Aura

  • Cold-start performance is a first-class concern
  • Short-lived requests are prioritized explicitly to provide some kind of determinism
  • Local, load-aware orchestration at the edge
  • No hidden control plane or opaque scheduling
  • Immutable artifacts with explicit lifecycles
  • Deploy ≠ activate
  • Built close to the kernel, not layered on generic servers
  • Regional and global function control

Version 1 goals

Version 1 focuses on delivering a complete, honest, self-hosted edge runtime and orchestrator.

Planned for v1:
  • Daemon configuration isolated from server configuration
  • Daemon key generation and secure storage
  • Lightweight CLI authentication
  • Encryption of function artifacts and configs at rest
  • Stabilized orchestration and scaling logic
  • Improved heuristics for short vs long-lived requests
  • JavaScript execution via QuickJS
  • Support for fetch, timers, and logging
  • Automatic function rollbacks based on predefined success matrix
Explicitly out of scope for v1:
  • WASM support (planned for later)
  • HTTP/1.x compatibility
  • Managed SaaS control plane
  • Opaque autoscaling or hidden scheduling logic

Design principles

  • Explicit over Implicit. - Every decision, lifecycle and scaling choice is observable.
  • Immutable artifacts. - Code, config, and deployment blobs are encrypted and versioned.
  • Edge-first orchestration. - Local, load-aware scaling without relying on a managed control plane.
  • Performance is measurable. - Cold-starts, request prioritization, and epoll strategies are tuned and observable.
  • Simple abstractions. - Minimal layers between the kernel and function execution.
  • Transparency over convenience. - No hidden defaults, no opaque scheduling.
  • Operator Control. - Deploy ≠ activate: explicit lifecycle transition.
  • Incremental correctness. - Early validation, precise errors, and small testable primitives
